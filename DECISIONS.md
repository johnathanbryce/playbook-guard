## Scored pipeline spec (2026-07-23)

- **Per-rule retrieval, not whole-contract prompting.** Each playbook rule is embedded (query built from `clause` + `preferred` + `hardStop`) and cosine-searched against that contract's chunks; Claude judges only the top-k passages. Why: keeps token cost bounded and makes the analysis input auditable — pgvector output *is* the LLM input.
- **Three verdicts only:** `compliant` / `deviation` / `not-addressed`. No fallback-tier scoring in MVP. Why: the fallbacks/hardStop in the playbook inform the judge's reasoning but the surfaced signal stays simple and testable.
- **Citation firewall is a hard gate, two stages.** (a) deterministic normalized-substring check of the cited quote against untouched `raw_text`; (b) a cheaper independent Claude pass confirming the quote supports the verdict. Labels: `verified` / `needs-review` / `fabricated`. Why: an ungrounded legal citation is worse than no tool; the quote must be proven present mechanically, not on the model's word.
- **Normalization, not fuzzy matching.** Firewall collapses whitespace and unifies quote/dash characters on both sides, then requires an exact substring. Why: fuzzy matching would let a paraphrase pass as grounded, defeating the firewall. Near-miss = `fabricated`.
- **`fabricated` is quarantined, not discarded.** Stored for debugging but excluded from the trustworthy result set and rendered separately. Why: we want to measure hallucination rate against the fidelity fixtures.
- **`escalate()` repurposed to department-email drafting.** Original stub intent ("route to stronger model / human") is dropped for MVP; confidence routing is handled by the `needs-review` label surfaced to the human reviewer. `escalate()` now drafts a text-only notification for a rule's `escalation.team`. Why: the product feature is department notification; there is no second flagger model in MVP.
- **Signature widening.** `retrieve(query, k)` gains a `contractId` filter (search must be scoped to one contract); `firewall(verdict, rawText)` takes the structured flag rather than an opaque verdict; `escalate` takes `(flag, rule, filename)`. Why: stubs were written before the per-contract, structured-flag design settled.
- **Caching.** Embeddings cached in Redis by `sha256(chunkText)`; flag results cached by `sha256(ruleJson + citedPassages)`. Firewall always re-runs (deterministic + cheap). Why: re-running an unchanged contract should be near-free, but the grounding check is the one thing we never skip.
- **Duplicate uploads reuse chunks but re-run analysis.** Dedup by `content_hash`; the playbook may have changed since last run. Why: chunks/embeddings are contract-derived (stable), verdicts are playbook-derived (not).
- **Embeddings via OpenAI `text-embedding-3-small`.** Why: Anthropic ships no embeddings API; the flagging/judge LLM is Claude, embeddings are OpenAI. Mixed-provider is deliberate, not an oversight.
- **Chunk on section boundaries, not fixed windows.** Split `raw_text` with a regex on section headers (e.g. `12.4`, `Section 12`, `ARTICLE X`) rather than fixed word-count + overlap. Why: legal meaning lives at the clause/section level; a fixed window can bisect a liability cap and destroy the passage the firewall must later match verbatim. Trade-off: a giant section may exceed ideal chunk size — acceptable for MVP, revisit if a section blows the embedding token limit.
- **ORM: Drizzle (confirmed).** Already scaffolded; schema.ts is the single source of truth for tables + the HNSW index, migrations generated from it. Why: no hand-written SQL init drift; typed queries into the pipeline.
- **Testing: Vitest.** Why: native to the Vite/TS/ESM stack, one runner for api and web, no Babel/ts-jest transform config. The firewall is the highest-value target — unit tests for grounded quote (verified), paraphrased quote (needs-review), and absent quote (fabricated) against `raw_text` fixtures.
- **Light middleware only.** CORS + JSON/body parsing + a single global error handler. No auth, rate limiting, or request logging in MVP. Why: single-user local demo; the engineering budget goes to the pipeline and firewall, not ops hardening.
- **Live SSE streaming.** Per-rule results stream to the frontend as each rule resolves (`rule` events + final `done`), rather than one batch response. Why: 6 sequential rule analyses each with a flag + firewall pass is slow; progressive fill keeps the reviewer engaged and shows the firewall doing its work.
- **Structured-result endpoint (`GET /analysis`) as the integration surface.** Add a machine-consumable JSON endpoint returning `{ contractId, playbookVersion, flags[], summary }` alongside the SSE stream. Both surfaces are fed by one shared `analyze(contractId)` pipeline: SSE for the live UI, JSON for outside consumers (GC's own systems). The frontend gets a live-SSE / raw-payload display toggle off the same data. Why: this turns the demo into a service — the UI is just one consumer of a real API, and the toggle makes that explicit to reviewers. SSE alone is awkward to parse from a non-browser integrator.
## Seeding + DB-backed playbook (2026-07-23)

- **Added a `playbooks` table for the envelope.** `playbook_rules` only holds rules, but `GET /playbook` must return the full envelope (name, version, agreementType, perspective, description) the frontend renders — and `version` is needed downstream as the analysis-cache key. Rather than a sentinel row in `playbook_rules` (violates "one row per rule") or hardcoding meta in the route (not really file→DB), added a `playbooks` table: `name`, `version`, `meta` jsonb (full envelope minus `rules`), `unique(name, version)`. `seed()` writes one playbook row + the rule rows; `GET /playbook` reconstructs `{ ...meta, rules }`. Why: DB-as-source-of-truth needs a home for playbook-level metadata; keeping meta as one jsonb blob is lossless and avoids enumerating columns that may change.
- **Seed is idempotent via upsert, not truncate+insert.** Playbook upserts on `(name, version)`, rules upsert on `rule_id`. Re-running `npm run db:seed` updates in place (verified: rule count stays 6, no dupes). A playbook *version* bump inserts a new `playbooks` row; `GET /playbook` serves the most recent by `created_at`. Why: safe to re-run in dev; version history is retained rather than clobbered.
- **Rules returned in insertion order.** `GET /playbook` orders rules by `playbook_rules.id` (serial) so display order matches the source JSON. Why: no explicit order column, and the playbook is authored in a deliberate order; serial insertion order preserves it for free.
- **Seed runs as an explicit CLI (`npm run db:seed`), not on API boot.** A `src/scripts/seed.ts` runner calls `seed()` and closes the pool. Why: seeding is a deliberate one-time/rare action; wiring it into startup would re-run on every container restart and muddy boot semantics. Tables are created with `npm run db:push` (drizzle-kit), no hand SQL.

- **Analysis-result cache for idempotency.** Cache the full analysis payload keyed on `(contract_hash + playbook_version)`, 24h TTL. A re-request against the same contract + same playbook version short-circuits the entire pipeline (retrieve/flag/firewall/escalate) and returns the cached result. Why: (1) perf — the pipeline is 6 rules × (flag + firewall) LLM calls; (2) correctness — it makes `GET /analysis` idempotent: same inputs → same output, which an integration surface needs. Keyed on playbook_version so a playbook bump invalidates cleanly; runs containing any `error` flag are not cached. This is separate from the embeddings cache (`sha256(chunkText)`), which we already had — the analysis-result cache was missing and is the one that makes the endpoint idempotent.

## Chunking + embedding (2026-07-23)

- **Chunk on ALL-CAPS section headers, one chunk per top-level section.** Regex `^(\d{1,2})\.[ \t]+([A-Z]…)$` (multiline) finds headers like `6. DATA PROCESSING AND PRIVACY`. A whole section (header + all its `6.x` sub-clauses) becomes exactly ONE chunk = ONE embedding — the section is never split. Why: the vital requirement is that a contract section stays intact as a single retrievable/embeddable unit that lines up 1:1 with a playbook clause; word-count windows with overlap would bisect a liability cap and scatter it across chunks. The `\.[ \t]+` (whitespace required after the dot) excludes `6.1`-style sub-clauses from matching, and requiring an uppercase title stops a wrapped clause line from being read as a boundary. Verified across all 4 contracts: 11–15 chunks each, largest ~350 tokens (well under the 8191 model limit, so no section ever needs splitting).
- **Chunk slices are verbatim substrings of raw_text (only outer whitespace trimmed).** Confirmed byte-faithful in the smoke test (`raw_text.includes(chunkText)` true for every chunk). Why: the citation firewall matches Claude's quote against untouched raw_text; if chunks were reformatted, a quote lifted from a chunk might not be found in the source and would be wrongly labeled `fabricated`.
- **Preamble captured as its own chunk.** Text before section 1 (title line, recitals) becomes a `Preamble` chunk. Why: never leave source text unretrievable, even though it rarely maps to a rule. A no-recognizable-sections document falls back to a single whole-document chunk.
- **Redis embedding cache lives in `embed()`, keyed `emb:<model>:<sha256(text)>`, 30d TTL.** `embed()` mgets all keys, sends only misses to OpenAI in one batched `embedMany`, pipelines the results back. Why here: embedding is a deterministic paid call — identical text → identical vector — so caching at this seam means a re-uploaded hash-dup contract and boilerplate shared across contracts (Definitions/License clauses are near-identical across the 4 test contracts) never pay twice. TTL is only to bound Redis growth, not for correctness (the vector never changes). Return order == input order so callers can zip vectors back to chunks.
- **Embed runs in the container, not the host.** `ai` + `@ai-sdk/openai` are in the container's node_modules (anonymous volume), absent on the host; `OPENAI_API_KEY` is durable in root `.env`, compose-interpolated, and confirmed present in the running container before any embed call. Why note it: host `npm run typecheck`/execution of embed will fail on missing deps — use `docker exec playbook-guard-api-1 …`.

## Ingestion + POST /contracts (2026-07-23)

- **Dedup key is sha256 of the UTF-8 raw_text, not the raw file bytes.** On a hit we reuse the existing contract + chunks and skip chunking and (paid) embedding entirely, returning `{contractId, chunkCount, deduped:true}`. Why raw_text and not bytes: the text is what we store and analyze; two uploads with identical text but trivially different encodings/BOM should still dedup. Trade-off: not the literal "sha256 of file bytes" the schema comment mentions — chose text-hash for stability against upload encoding noise.
- **Upload = multer memoryStorage, 5 MB cap, multipart field `file`.** `.txt`-only, accepted if mimetype is `text/plain` OR the filename ends `.txt` (browsers/curl set inconsistent mimetypes). Empty/whitespace file -> 400; non-txt -> 415; no file -> 400. Why memory not disk: contracts are small and we only need the buffer as a string; no temp-file cleanup.
- **201 vs 200 status split.** Fresh ingest returns 201 (created), a dedup hit returns 200 (nothing created). Why: lets an integrator distinguish "new work happened" from "already had it" without parsing the body.
- **Tests are a host-only devDependency; tsconfig excludes `**/*.test.ts`.** Vitest lives in host node_modules, not the container. Added `"exclude": ["**/*.test.ts"]` so the container's production `tsc --noEmit` doesn't fail on missing vitest types. Why: keeps the runtime image free of test deps; tests run on the host via `npm test`. Cost: `tsc` no longer type-checks test files — acceptable, vitest still executes them.

## Retrieval — per-rule pgvector search (2026-07-23)

- **`retrieve()` is a generic semantic-search primitive, not rule-aware.** Signature finalized as `retrieve(query: string, { contractId, k = 3 })` (the stub was `retrieve(query, k)`; `contractId` scope was pre-flagged). Rule→query construction lives in a separate `rule-query.ts` (`ruleToQuery(rule)`), so the retriever has zero coupling to the playbook shape and stays reusable. Why: keeps the vector-search seam clean; `flag`/`analyze` compose `retrieve(ruleToQuery(rule), …)`.
- **Query text = `clause` + `preferred`.** The positive, topical description of a rule. Why not `+ hardStop`: the eval showed clause+preferred already ranks every expected section #1 with wide margins, and violation phrasing risks diluting the topical signal. Why not clause-only: richer text is more robust when a contract renames/merges section headers.
- **Cosine distance via drizzle `cosineDistance` (`<=>`), ordered ascending, limit k.** Matches the `chunks` HNSW `vector_cosine_ops` index so the index is used. `similarity = 1 - distance` returned alongside for downstream relevance judgment. `distance` selected as `sql<number>` and `Number()`-coerced (float8).
- **Top-3, scores returned, NO threshold in retrieve.** `retrieve` always returns the k nearest with scores; the `flag` step decides "not-addressed" from low similarity. Why: keeps a tunable magic number out of the retriever and lets the LLM judge see the candidates + confidence.
  - **[AMENDED] (2026-07-23, prompted by John) — add similarity as a *signal*, not a hard gate.** John challenged the flat "no threshold" reading, asking whether we need a cosine cutoff (he floated 0.6 → 0.45) to detect contracts that don't match the playbook. On review we converged (this was a convergence, not an `[OVERTURN]` — the change aligns with Claude's revised recommendation): a **high** gate (~0.6) is rejected because it sits inside our measured *noise* band (eval on contract #1: real matches 0.72–0.85, unrelated 0.44–0.60, genuinely-relevant secondary sections 0.58/0.69), so it would risk silently reclassifying real deviations as `not-addressed` — the worst error for this product; cosine from `text-embedding-3-small` is also uncalibrated and we've only measured one high-fidelity contract. Instead, similarity is used three ways, none of which drop chunks from `retrieve`: (1) **low floor short-circuit (~0.35, tunable)** — if a rule's *top* chunk is below it, mark `not-addressed` without an LLM call (cheap omission/off-topic detector); (2) **pass top-1 similarity to the flag judge** so its `not-addressed` call is informed, not overridden, by weak retrieval; (3) **per-contract coverage metric** (count rules with a confident on-topic match ≥~0.7) so a wholly non-matching upload reads as e.g. "0/6 covered". Semantics: the floor is per-rule; a contract missing all six clauses → all six hit the floor → 0/6 coverage. Caveat recorded: 0.35 catches *clear* misses only (semantic search always returns *some* nearest chunk); borderline omissions are backstopped by the LLM judge + the ≥0.7 coverage bar. **To tune:** measure floor/coverage numbers against all 4 fixtures + a deliberately off-topic fixture (NDA/lease) before locking values — don't guess. `retrieve()` itself is unchanged (still returns top-k + scores, no drop); the floor/coverage live in `flag`/`analyze`.
- **Verified empirically, not just typechecked.** A throwaway eval over contract #1 confirmed all 6 rules retrieve their expected section at rank #1 (sim 0.72–0.85), and rules spanning two sections (service-levels-termination, auto-renewal-pricing) surface their 2nd section at #2 — validating k=3 before building `flag`.
- **[AMENDED] (2026-07-23, evidence-driven — not an `[OVERTURN]`, John asked to *see the numbers before locking the floor*) — floor + coverage tuned against the full corpus + an off-topic fixture; the floor's role is corrected.** Added a deliberately off-topic mutual NDA at `data/fixtures/offtopic-nda.txt` (ours; original `data/contracts/` corpus left pristine per John), ingested all 4 real contracts + the NDA, and measured top-1 similarity for every rule × contract. Raw top-1 matrix:
  ```
  rule                          high   medium   low    partial  offtopic
  limitation-of-liability      0.817   0.840   0.797   0.807    0.453
  indemnification              0.767   0.747   0.700   0.518    0.425
  data-processing-privacy      0.845   0.780   0.632   0.454    0.396
  ai-training-output-rights    0.727   0.662   0.688   0.448    0.371
  service-levels-termination   0.721   0.704   0.743   0.811    0.477
  auto-renewal-pricing         0.765   0.772   0.750   0.770    0.351
  coverage ≥0.70:               6/6     5/6     3/6     3/6      0/6
  ```
  **Correction to the prior `[AMENDED]` claim that the ~0.35 floor detects a wholly-non-matching upload — it does NOT.** The off-topic NDA (which genuinely contains none of the six clauses) scores 0.351–0.477 top-1 on *all six* rules — every one clears 0.35. So the floor is essentially inert on real legal prose and cannot be the off-topic gate. Two bands overlap and forbid any clean per-rule cutoff: genuine *not-addressed* in a real contract (`partial`'s omitted indemnification/DPA/AI) lands at **0.448–0.518**, while off-topic noise lands at **0.351–0.477**. A cutoff high enough to catch off-topic would wrongly short-circuit real-but-weak matches and steal the not-addressed call from the LLM. **Locked values:** (1) **floor = 0.35**, reframed as a *degenerate-retrieval backstop* only (fires on empty/garbage retrieval; sits below the entire off-topic band and below the weakest real match at 0.448, so on this corpus it ~never fires — correct for a backstop); the LLM judge owns compliant/deviation/not-addressed above it. (2) **coverage bar = 0.70** is the *actual* wholly-non-matching-upload detector — clean separation: off-topic 0/6 vs every real contract 3–6/6. Coverage lives in `analyze`; the floor lives in `flag`.

## Claude SDK choice (2026-07-23)

- **Claude calls use the official `@anthropic-ai/sdk`, not the Vercel AI SDK.** A live smoke test against `claude-sonnet-5` proved the AI SDK path is broken here: `@ai-sdk/anthropic@1.2.12` (the only version compatible with the repo's `ai`@4) sends a `temperature` parameter, which Sonnet 5 rejects with `400 "temperature is deprecated for this model"` (sampling params were removed on that model). The version that fixes it (`@ai-sdk/anthropic@4`) requires `ai`@5 — a full AI-SDK major upgrade that would also churn the working `embed.ts`. So: OpenAI embeddings stay on `ai` + `@ai-sdk/openai` (untouched), and Claude flagging + firewall run on the official `@anthropic-ai/sdk` (0.113.0, added to package.json + lockfile). Why accept a mixed-SDK setup: smaller blast radius than an AI-SDK v5 migration, the official SDK is the canonical/stable path with no phantom sampling params, and each half is independently robust. `@ai-sdk/anthropic` stays pinned `^1.2.12` but is effectively unused.
- **Smoke test before building the firewall.** Confirmed connectivity with a disposable write-use-destroy script (`claude-sonnet-5` → "PONG", `stop_reason: end_turn`, 35 in / 5 out tokens, ~1.3s) before wiring `flag`/`firewall`. Pre-flight also confirmed `ANTHROPIC_API_KEY` is durable in root `.env`, compose-interpolated, and present in the running container.

## flag service (2026-07-23)

- **Structured output via forced tool-use, not free-text parsing.** Both `flag` (Sonnet 5) and `firewall` (Haiku) get their structured result by defining a single tool (`record_verdict` / `record_check`) with a JSON `input_schema` and setting `tool_choice: {type:"tool", name:...}`. The model is forced to emit exactly that shape; we read `content.find(b => b.type==="tool_use").input`. Why: robust across SDK versions (0.113.0 predates the newer `messages.parse`/`output_config`), and forcing the tool guarantees a parseable object with no prose to strip. Consequence: **no extended thinking** on these calls — the API disallows `thinking` together with a forced `tool_choice`. Acceptable: each call is a single bounded judgment over a handful of passages, not a multi-step reasoning task. No `temperature` is sent (the exact param that broke the AI SDK path).
- **`flag` output shape is designed for the firewall.** `citedText` MUST be a verbatim substring of a retrieved passage (and therefore of `raw_text`); the prompt forbids paraphrase/ellipsis. `not-addressed` forces `citedText` to `""` so the firewall never grounds a stray quote. `flag` also returns `topSimilarity` (analyze applies the 0.70 coverage bar), `shortCircuited`, and the retrieved `passages[]` for audit.
- **Floor short-circuit lives in `flag`, verified in practice.** Live 6-case run across the fidelity ladder + the NDA: the 0.35 floor **never fired** (off-topic sims 0.351–0.477 all clear it) — the LLM correctly returned `not-addressed` for every off-topic and omitted-clause case, and all cited compliant/deviation verdicts were GROUNDED by a normalized-substring preview. This empirically confirms the retrieval `[AMENDED]` decision: floor = inert backstop, LLM owns not-addressed.

## Citation firewall (2026-07-23)

- **Deterministic-first, two independent stages.** (a) normalized-substring grounding of `citedText` against untouched `raw_text`; only if that passes do we spend (b) an LLM call. Labels: `verified` (a✓+b✓), `needs-review` (a✓ but judge not confident), `fabricated` (a✗ — quarantined, b never runs). Why deterministic-first: the grounding check is the firewall's teeth and is free; an ungrounded quote is fabricated regardless of what any model thinks, so there's no point asking the judge about it.
- **Judge model = `claude-haiku-4-5` (cheaper tier), John's call.** Realizes the "two Claude tiers" decision: flagger = Sonnet 5, firewall judge = Haiku 4.5. The judge is a narrow, high-volume confirmation ("does this exact quote support this verdict?"), which the cheaper model handles — and it must be a *different, independent* pass from the flagger, not the same model re-grading itself.
- **Normalization is deliberately minimal: collapse whitespace + unify smart quotes/dashes to ASCII, then exact substring.** This absorbs only cosmetic differences (line wrapping, curly vs straight quotes/dashes) — anything more (a paraphrase, an inserted ellipsis, a dropped word) changes the string and fails → `fabricated`. Fuzzy matching is explicitly rejected: it would let a paraphrase pass as grounded, defeating the firewall. `normalizeForMatch` is exported for the Vitest suite (the highest-value test target: grounded / off-point / fabricated fixtures).
- **`not-addressed` (empty citation) → `verified` with `grounded:false`.** A verdict that makes no textual claim has nothing to fabricate, so the surfaced result is trustworthy (`verified`); `grounded:false` records that there is no verified quote to display. The UI shows the citation chip only when `grounded` is true. This keeps every flag labelled with one of the three statuses without a fourth state.
  - **[OVERTURN] (2026-07-23, John) — not-addressed → `not-applicable`, NOT `verified`; a 4th status added.** Claude recommended (and shipped) labelling an empty-citation not-addressed flag `verified`, arguing "nothing to fabricate → trustworthy." John overruled: `verified` with `grounded:false` is self-contradictory, and — the real problem — it **inflates `summary.firewall.verified` with rules that had no citation to check**. The firewall grades *citations*; a not-addressed flag has none. New semantics: empty `citedSpan` on a **not-addressed** verdict → **`not-applicable`** (its own summary bucket `notApplicable`); empty `citedSpan` on a **compliant/deviation** verdict — a claim asserted with no evidence, the *dangerous* case — → **`needs-review`**. `verified` now strictly means "had a citation, confirmed real + supportive." `FirewallStatus` is now a 4-value union; the `summary.firewall` buckets (verified/needsReview/fabricated/notApplicable) each get exactly one increment per flag, so they sum to `ruleCount`.
- **Verified in isolation with synthetic Flags.** Drove one case to each label (grounded-supporting → verified; grounded-but-off-point Definitions quote → needs-review; invented quote → fabricated; empty citation → verified). Notably the judge is genuinely skeptical: a *truncated* cap quote that didn't establish mutuality was correctly held at `needs-review` until given the complete supporting text — the independent pass is not a rubber stamp.

## analyze pipeline — one core, two surfaces (2026-07-23)

- **ONE core function `analyze(contractId, onRule?)`; two routes consume it.** The pipeline (per rule: retrieve → flag → firewall → escalate) lives here exactly once. It returns the full structured aggregate AND calls `onRule(ruleResult)` per rule as each resolves. `GET /stream` (SSE demo) passes an `onRule` that writes an event; `GET /analysis` (integration JSON) passes none and awaits the aggregate. **Hard constraint (John):** the JSON endpoint must be a real server-side call to `analyze()`, NOT SSE events accumulated client-side — otherwise the "it's a callable API" story is fake.
- **`onRule` fires in every path, including warm cache hits.** On a cache hit, `analyze()` still replays each cached rule through `onRule` (then the route sends `done`), so the streaming UX is identical whether the cache is warm or cold; `/analysis` just returns the cached aggregate. The replay lives inside the core, not the route.
- **Result cache wraps the return value.** Keyed `analysis:<contentHash>:<playbookVersion>`, 24h TTL, Redis JSON. Compute → emit per rule → cache the aggregate → return it. `cached:true` + a stable `generatedAt` on warm hits makes idempotency visible in the payload. Only complete runs are cached — a rule failure throws and aborts before the cache write, so no partial is ever persisted. (Per-rule error resilience — emit an `error` rule and continue — deliberately deferred; out of scope for the core.)
- **Escalation gated on the firewall.** analyze never drafts an escalation for a `fabricated` flag — an ungrounded citation must not reach a department. Escalations are embedded per-rule AND collected in top-level `escalations[]` for the email panel (same object).
- **Field naming locked (John): `citedSpan`, not `citedText`; `firewall` stays a nested object, status at `firewall.status`.** Renamed end-to-end (Flag type, firewall, escalate, analyze, tests) to match the vocabulary John uses across DB/API — one term, no internal/external seam. NO duplicate flat `firewallStatus` (one source of truth avoids drift). The nested object is also the better integration payload: a consumer gets grounding + reasoning, not a bare string. NOTE: there is currently **no `analyses` table** — the aggregate is Redis-cached JSON; `cited_span`/`grounding_status` are the column names to use IF analyses are later persisted to a table. `firewall.status` IS the grounding status (the firewall produces it).
